Contributing

Documentation-first contribution workflow with scan validation and explicit uncertainty handling.

Contribution Principles

  • Do not assume behavior that is not verified in code.
  • If uncertain, mark with a visible SCAN NEEDED note and exact follow-up target.
  • Keep docs synchronized with API contract and deployment config changes.
  • Preserve existing style and keep changes minimal in code PRs.

Branch & PR Flow

  1. Create a branch from default branch.
  2. Implement code and update affected docs page(s).
  3. Run local checks/build.
  4. Open PR with testing evidence and screenshots for UI/docs changes.
SCAN NEEDED: Repository-specific branch naming and PR template standards (not found in scanned files).

Documentation QA Checklist

  • All links in sidebar resolve.
  • Mermaid diagrams render without syntax errors.
  • No orphan SCAN NEEDED sections for verified facts.
  • No stale endpoint names or env keys.
  • GitHub Pages deployment workflow passes on default branch.

Code QA Checklist

  • Frontend builds and starts locally.
  • Backend starts and /health responds.
  • Critical flows tested: auth, evaluate, session plan, breathwork protocols.
  • No secrets committed in source-controlled files.
